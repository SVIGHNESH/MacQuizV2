Section 01

What v2 demands from hosting

The cheapest deployment is decided by the architecture, not by shopping. Four properties of MacQuiz v2 disqualify most "free hosting" offers before price is even discussed.

RequirementFrom SDD-001What it rules out
Long-lived WebSocketsRealtime gateway: live dashboard, attempt channel, kick delivery (< 1 s)Pure serverless functions; free tiers that cap connection duration
Always-on schedulerQuiz open/close jobs and per-attempt deadline timers fire on server timeFree plans that sleep on idle (Render free spins down after 15 min - a sleeping scheduler misses starts_at)
No cold starts during a live window1,000+ students start in the same minute; kick and auto-submit are time-criticalScale-to-zero platforms; anything that needs 30-60 s to wake
Redis pub/sub + queueEvent fan-out between API and gateway; BullMQ jobs for imports, grading, rollupsHTTP-only "serverless Redis" free tiers with tight command/bandwidth caps

The one thing v2 does not demand is horizontal scale on day one. SDD-001's capacity math (Section 15) put peak load at roughly 200 DB writes/s and 250 fan-out events/s. That is a workload for one modest machine, and a single machine is exactly where hosting is cheapest.

Section 02

Strategy: one VM, everything on it

The whole platform - API monolith, realtime gateway, workers, PostgreSQL, Redis, and the reverse proxy - runs as one Docker Compose stack on a single always-free VM. Free managed services are used only where they are genuinely better than self-hosting at this scale: DNS/CDN, object storage, email, and CI.

  • Primary host: Oracle Cloud "Always Free" ARM VM. Up to 4 Ampere OCPUs, 24 GB RAM, 200 GB block storage, 10 TB/month egress - permanently free, not a trial. This is more machine than the SDD-001 capacity model needs by an order of magnitude.
  • Fallback host: Hetzner CX22 (2 vCPU / 4 GB, ~€3.8 ≈ ₹390/month). Oracle free-tier capacity in popular regions (including Mumbai) can take days of retrying to obtain. The fallback exists so that "free" never blocks the launch; the stack is identical either way.
  • Colocated Postgres and Redis. At this scale, running them beside the app removes network latency from the autosave path (helps the < 300 ms p95 target), removes two external dependencies, and removes two bills. The trade is that backups become our job - Section 8 handles that.
  • Cloudflare free in front. DNS, TLS at the edge, CDN for the static frontend, WebSocket proxying (free plan supports it), and absorption of casual abuse before it reaches the VM.
Design decision

SDD-001 chose a modular monolith so that everything communicates through Redis and a job queue inside one deployable. That decision is what makes the $0 deployment possible: one process tree, one VM, no per-service hosting fees, and extraction later remains a deployment change.

Section 05

The managed-split path (and why it lost)

The popular free-tier recipe - app on Render/Railway, Postgres on Neon or Supabase, Redis on Upstash - was evaluated and rejected as the primary path. It is a fine pattern for a CRUD app; MacQuiz is not one.

PieceFree-tier offerWhy it breaks MacQuiz v2
Render (free web service)750 h/month, sleeps after 15 min idleA sleeping instance misses open_quiz/close_quiz jobs and cold-starts ~1 min into the go-live herd. Disqualifying.
Railway / Fly.io$5 one-time or usage creditNot actually free past the first weeks; fine products, but then Hetzner is cheaper and simpler.
Neon Postgres0.5 GB, scale-to-zero computeCompute suspends when idle; first autosave after wake eats a multi-second cold start against a < 300 ms p95 target. Also adds WAN latency to every write.
Upstash Redis500k commands/monthOne live quiz (heartbeats every 10 s, autosaves, pub/sub fan-out) burns the monthly command budget in days; per-command pricing inverts the design's "chatty Redis is fine" assumption.
Pusher / Ably free100-200 concurrent connectionsOne 500-student quiz exceeds the cap; per-connection pricing punishes exactly our load shape.
The pattern

Every managed free tier is shaped for apps that are mostly idle and lightly connected. MacQuiz is the opposite: idle for days, then intensely busy and massively connected for one hour. A VM does not care about that shape; per-request and per-connection pricing does.

Section 06

The Compose stack

One file defines the whole backend. The app image is built once in CI and pulled here; Postgres and Redis pin major versions and persist to named volumes on the VM's block storage.

docker-compose.yml (abridged)
services:
  caddy:
    image: caddy:2
    ports: ["80:80", "443:443"]
    volumes: [./Caddyfile:/etc/caddy/Caddyfile, caddy_data:/data]

  app:                            # API monolith + realtime gateway (/ws)
    image: ghcr.io/ORG/macquiz-app:${TAG}
    env_file: .env.production
    depends_on: [postgres, redis]
    restart: unless-stopped
    deploy: { resources: { limits: { memory: 4g } } }

  worker:                         # BullMQ: scheduler, grading, imports, rollups
    image: ghcr.io/ORG/macquiz-app:${TAG}
    command: node dist/worker.js
    env_file: .env.production
    depends_on: [postgres, redis]
    restart: unless-stopped

  postgres:
    image: postgres:16
    volumes: [pg_data:/var/lib/postgresql/data]
    environment: { POSTGRES_DB: macquiz }
    restart: unless-stopped

  redis:
    image: redis:7
    command: redis-server --appendonly yes   # queue jobs survive restarts
    volumes: [redis_data:/data]
    restart: unless-stopped

volumes: { pg_data: {}, redis_data: {}, caddy_data: {} }
  • API and realtime gateway share one process at this tier. SDD-001 keeps them separable; here, separation would only add a container without adding capacity. Splitting is a Compose edit when the time comes (Section 11).
  • Redis AOF persistence is on so delayed jobs (quiz open/close, deadline timers) survive a restart. Belt and braces: the worker also re-scans Postgres for due-but-unfired transitions at boot - the lazy state validation SDD-001 already requires.
  • Firewall: only 80/443 open to the world (and Cloudflare IPs at that), SSH on a non-standard port with key-only auth. Postgres and Redis are never exposed; they live on the Compose network.

Section 07

CI/CD on free minutes

  1. On pull request: lint, typecheck, unit + integration tests (Postgres/Redis as GitHub Actions services). Public repos get unlimited free minutes; private repos get 2,000/month - enough for this pipeline many times daily.
  2. On merge to main: build the ARM64 image (Actions ARM runners are free for public repos; otherwise buildx cross-compile), push to GHCR, then SSH to the VM: TAG=sha docker compose pull app worker && docker compose up -d app worker.
  3. Migrations run in the app's entrypoint before it accepts traffic, and deploys are refused by a pre-deploy check while any quiz is live - a self-imposed deploy window, which costs nothing and prevents the worst incident this platform could have.
  4. Frontend: Cloudflare Pages builds from the same repo on push; preview deployments per PR are free.

Section 08

Backups, monitoring, ops

Backups - the non-negotiable part of self-hosted Postgres

  • Nightly pg_dump (custom format, compressed) from a cron container, uploaded to a versioned R2 bucket. Attempt data for a whole academic year fits in single-digit GB - well under the free 10 GB.
  • Retention: 7 daily + 8 weekly dumps, pruned by the same job. R2 object versioning protects against a bad prune.
  • Restore drill once per term: pull the latest dump into a scratch container and run the smoke tests against it. An untested backup is a hope, not a backup.
  • Extra belt during exams: a pre-quiz-window dump is triggered automatically by the scheduler when any quiz enters scheduled for the same day.

Monitoring on $0

  • UptimeRobot pings /healthz (checks DB + Redis + queue depth) every 5 min from outside, alerting by email/Telegram - catches the "VM died" class of failure that self-hosted monitoring cannot see.
  • Grafana Cloud free tier receives OpenTelemetry metrics and logs from the app: autosave p95, WS connection count, queue lag, violation/kick event rates. The SDD-001 observability intent survives at $0, just with 14-day retention.
  • Watchtower is deliberately absent: images update only via the deploy pipeline, never automatically under a live quiz.

Section 09

Does free actually fit?

Re-running SDD-001's capacity model against the free VM:

Load driverPeak estimateFree VM headroom
Autosave writes2,000 students × 1/10 s ≈ 200 writes/s, single-row PK upsertsPostgres on 4 ARM cores + NVMe-backed volume handles thousands/s
WebSocket connections~2,000 student sockets + dashboards~0.5-1 GB RAM in Node; VM has 24 GB
Event fan-out~250 coalesced events/s through Redis pub/subRedis pub/sub is comfortable to ~100k msg/s on one core
Go-live herd1,000 attempt-starts in ~60 s; snapshot served from Redis cache~17 starts/s of small transactions - negligible
Grading spike at close2,000 attempts × ~30 questionsQueue-absorbed; minutes of one worker core, latency-insensitive

The free tier is not the compromise in this plan - it is over-provisioned roughly 10× for the stated load. The genuine compromises are operational: single-region, single-node, self-managed backups, and a deploy freeze during live windows instead of zero-downtime rollouts.

TierComputeEverything elseMonthly total

Recommendation

Provision Tier 0 now (Oracle sign-up and ARM capacity hunting can run in the background), launch on Tier 1 if the VM is not secured by launch week, and treat Tier 2 as the first spend once real exams depend on the platform. The stack is byte-identical across all three.

Section 11
