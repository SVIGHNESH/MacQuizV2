- Milestone 5 - Realtime (FR-7): server/internal/realtime/ is a doc-comment stub. DONE so far: the GET /quizzes/:id/live roster snapshot (docs/05 section 4) - one row per assigned student collapsed to their latest attempt, owner-or-admin auth via ActionQuizWatchLive, answered_count/question_count/max_score from the pinned snapshot, server_time origin; proven by the TestLiveRosterE2E integration suite and OpenAPI v0.9.0 (getLiveRoster + LiveRosterRow/LiveRosterResponse). Also DONE: the attempt_events source-of-truth layer (docs/05 sections 1-2) - every lifecycle write now appends its typed event row in the same transaction as the state change ("persist first"): attempt.started (fresh start only, resume emits none), attempt.progress (answered_count only; current_question stays null - no server cursor over REST), attempt.submitted for all three kinds (manual funnel + the auto/forced batch sweep, which was rewritten to one transaction with UPDATE ... RETURNING so only rows it actually flipped emit), and attempt.graded (score, once per grade); proven by TestEventFlowE2E. Also DONE: the Redis pub/sub publish brick (docs/05 section 1 "publish second") - realtime.Publisher (go-redis) relays each committed event to quiz:{quiz_id}:events after its transaction commits, injected as the small attempt.EventPublisher interface into the attempt Service (serve-side started/progress/submitted) and threaded into the worker's SweepDueAttempts/GradeSubmitted (worker-side batch submitted + graded), best-effort by contract (a publish failure logs and drops; the live snapshot reconciles), proven by TestEventPublishE2E (capture-fake: right channel/payload, publish-only-after-commit, no publish on rollback/no-flip) plus a Redis-gated TestPublisherLiveSmoke. Also DONE: attempt.progress coalescing (docs/05 section 5) - attempt.ProgressCoalescer, a leading-edge EventPublisher decorator wrapped around the serve-side publisher at the composition root (main.go), relays at most one attempt.progress per 2s per attempt (keyed by attempt_id, one attempt = one student) while every other event type passes straight through, so a 500-student quiz now stays inside the doc's ~250 events/s budget; coalescing is delivery-only (SaveAnswer still appends every progress row, so the append-only log and the reconciling snapshot stay complete), memory is bounded by an amortized sweep-on-write (no janitor goroutine), and the worker is left unwrapped since it emits no progress; proven by the injectable-clock unit test TestProgressCoalescer (rapid burst -> 1 relay, window boundary admits the next, per-attempt independence, non-progress always passes, sweep bounds the map). Also DONE: the WebSocket monitor gateway (docs/05 section 1 hop 3 + section 3) - realtime.Gateway mounts GET /ws/quizzes/{id}/monitor, authenticates via the same authusers.RequireAuth (the access-token cookie the browser sends on a WS handshake, since browsers cannot set an Authorization header), gates on the coarse teacher-or-admin requireStaff (403 for a student) then runs the one subscribe-time Can(ActionQuizWatchLive) check the docs mandate (404 for a non-owning teacher or an unknown quiz - existence never leaked, mirroring GET /quizzes/:id/live via a new narrow quiz.Service.OwnerOf), subscribes to quiz:{id}:events through a Subscriber interface (redis-backed NewRedisSubscriber built with go-redis default timeouts, split from the publisher's fast-fail client since a pub/sub receive blocks), and fans each envelope out to the socket with a 30s keepalive ping. The socket lifetime is deliberately detached from the request context and bound to the serve process context, so the root router's 30s middleware.Timeout (now scoped to the REST group only, not /ws - chi's Timeout also writes a 504 on return, which on a hijacked socket errors) can never guillotine a live monitor and every socket closes on shutdown. coder/websocket added as the WS library with OriginPatterns wired from a new MACQUIZ_WS_ALLOWED_ORIGINS config (same-origin only by default). authusers.WithActor added as the write-side counterpart to ActorFrom (RequireAuth now uses it; also the injection point for the WS authenticator and tests). Proven by fake-Subscriber unit tests (happy-path relay, socket-outlives-request-timeout regression guard mutation-verified, shutdown-closes-socket, non-owner 404, unknown-quiz 404, bad-uuid 404, requireStaff role matrix, upgrade-through-wrapped-ResponseWriter) plus a Redis-gated TestGatewayLiveSmoke exercising the real Publish->subscribe->socket path. Remaining: the attempt:{id} and user:{id}:notify channels (kick delivery, quiz.extended banners, assignment notifications), heartbeat/disconnect tracking (attempt.disconnected/reconnected), wiring the events into the live snapshot's current_question, and the teacher live dashboard UI.

Smaller design-doc items with no home yet

- Audit log read side - writes happen everywhere, but the admin-facing GET /audit endpoint is missing.
- Live-quiz teacher controls - extend ends_at, force-close early, and archive (the fifth lifecycle state) have no endpoints; only the automatic scheduler close exists.

In short: the server-authoritative core (identity, authoring, scheduling, attempts, grading) is done and solid,but the platform is not yet usable by a student or admin through the browser. The natural next steps in plan order are finishing Milestone 4 (results release + the web attempt player, which is also the exit criterion's user-visible half), then realtime, guardrails/kick, import, and analytics.
