- Milestone 5 - Realtime (FR-7): server/internal/realtime/ is a doc-comment stub. DONE so far: the GET /quizzes/:id/live roster snapshot (docs/05 section 4) - one row per assigned student collapsed to their latest attempt, owner-or-admin auth via ActionQuizWatchLive, answered_count/question_count/max_score from the pinned snapshot, server_time origin; proven by the TestLiveRosterE2E integration suite and OpenAPI v0.9.0 (getLiveRoster + LiveRosterRow/LiveRosterResponse). Also DONE: the attempt_events source-of-truth layer (docs/05 sections 1-2) - every lifecycle write now appends its typed event row in the same transaction as the state change ("persist first"): attempt.started (fresh start only, resume emits none), attempt.progress (answered_count only; current_question stays null - no server cursor over REST), attempt.submitted for all three kinds (manual funnel + the auto/forced batch sweep, which was rewritten to one transaction with UPDATE ... RETURNING so only rows it actually flipped emit), and attempt.graded (score, once per grade); proven by TestEventFlowE2E. Also DONE: the Redis pub/sub publish brick (docs/05 section 1 "publish second") - realtime.Publisher (go-redis) relays each committed event to quiz:{quiz_id}:events after its transaction commits, injected as the small attempt.EventPublisher interface into the attempt Service (serve-side started/progress/submitted) and threaded into the worker's SweepDueAttempts/GradeSubmitted (worker-side batch submitted + graded), best-effort by contract (a publish failure logs and drops; the live snapshot reconciles), proven by TestEventPublishE2E (capture-fake: right channel/payload, publish-only-after-commit, no publish on rollback/no-flip) plus a Redis-gated TestPublisherLiveSmoke. Also DONE: attempt.progress coalescing (docs/05 section 5) - attempt.ProgressCoalescer, a leading-edge EventPublisher decorator wrapped around the serve-side publisher at the composition root (main.go), relays at most one attempt.progress per 2s per attempt (keyed by attempt_id, one attempt = one student) while every other event type passes straight through, so a 500-student quiz now stays inside the doc's ~250 events/s budget; coalescing is delivery-only (SaveAnswer still appends every progress row, so the append-only log and the reconciling snapshot stay complete), memory is bounded by an amortized sweep-on-write (no janitor goroutine), and the worker is left unwrapped since it emits no progress; proven by the injectable-clock unit test TestProgressCoalescer (rapid burst -> 1 relay, window boundary admits the next, per-attempt independence, non-progress always passes, sweep bounds the map). Also DONE: the WebSocket monitor gateway (docs/05 section 1 hop 3 + section 3) - realtime.Gateway mounts GET /ws/quizzes/{id}/monitor, authenticates via the same authusers.RequireAuth (the access-token cookie the browser sends on a WS handshake, since browsers cannot set an Authorization header), gates on the coarse teacher-or-admin requireStaff (403 for a student) then runs the one subscribe-time Can(ActionQuizWatchLive) check the docs mandate (404 for a non-owning teacher or an unknown quiz - existence never leaked, mirroring GET /quizzes/:id/live via a new narrow quiz.Service.OwnerOf), subscribes to quiz:{id}:events through a Subscriber interface (redis-backed NewRedisSubscriber built with go-redis default timeouts, split from the publisher's fast-fail client since a pub/sub receive blocks), and fans each envelope out to the socket with a 30s keepalive ping. Proven by fake-Subscriber unit tests (happy-path relay, socket-outlives-request-timeout regression guard mutation-verified, shutdown-closes-socket, non-owner 404, unknown-quiz 404, bad-uuid 404, requireStaff role matrix, upgrade-through-wrapped-ResponseWriter) plus a Redis-gated TestGatewayLiveSmoke exercising the real Publish->subscribe->socket path. Remaining: the attempt:{id} and user:{id}:notify channels (kick delivery, quiz.extended banners, assignment notifications), heartbeat/disconnect tracking (attempt.disconnected/reconnected), wiring the events into the live snapshot's current_question, and the teacher live dashboard UI.
- Milestone 6 - Guardrails and kick (FR-10, FR-11): DONE so far: the kick brick (docs/06 section 4) - POST /attempts/:id/kick {reason} removes a student from a live attempt with owner-or-admin auth (ActionAttemptModerate, decided against the attempt's quiz owner so a non-owning teacher and an unknown attempt both read 404; a student is 403 via a requireStaff group added to the attempt mount; the reason is required, else 422). One transaction flips the attempt to the terminal 'kicked' state (status='kicked', submit_kind='kicked', kicked_by, kick_reason, submitted_at), appends the attempt.kicked event (new eventKicked + kickedPayload{kicked_by, reason}), writes the audit_log row, and enqueues grading; the Redis relay fires the attempt.kicked delta once after commit. Enforcement is the status flip, not the socket - the existing writableForUpdate/submit gates already reject 'kicked' with 409 ATTEMPT_KICKED, so a kicked student's autosave and submit are both refused. Idempotent: a repeat kick, or one that lost the race to a submit, no-ops (no second event/publish/audit). "Kicked work is graded, not discarded" (docs/06:80): gradeOne + GradeSubmitted now grade the two ungraded terminal states - 'submitted' and 'kicked' - both advancing to 'graded', so 'graded' stays the single "grading landed" signal every results read gates on (GET /attempts/:id/result); the kick is preserved by submit_kind='kicked', off which the live roster derives its permanent 'kicked' state (rosterState now reads submit_kind), so the student shows 'kicked' both before and after grading. Proven by TestKickFlowE2E (auth matrix, flip+event+publish+audit, lockout, idempotence, grade-keeps-kicked, roster, admin kick, submit-race no-op); OpenAPI bumped to v0.10.0 (kickAttempt) with the web client regenerated. Also DONE: the re-admission brick (docs/06 section 4:81) - POST /attempts/:id/readmit {reason} grants a kicked student one fresh attempt slot without resurrecting the kicked attempt ("re-admission is a new attempt, not a resurrection"). Same owner-or-admin auth as kick (ActionAttemptModerate via the shared moderatableForUpdate, so a non-owning teacher and an unknown attempt both read 404, a student is 403, a blank reason is 422); the target must be a kicked attempt, else 409 ATTEMPT_NOT_KICKED (new CodeAttemptNotKicked). The grant is a single new attempts.readmitted_at timestamptz column (migration 00005) set only WHERE status='kicked' AND readmitted_at IS NULL, so a repeat readmit (or a concurrent double click) grants no second slot and writes no second audit row - the marker IS the grant record, no counter to drift. Start now reads count(readmitted_at) in its existing per-student count query and checks used >= max_attempts + granted, so each readmit nets exactly one fresh attempt while the kicked row keeps counting (it "stays in the record"). The kicked attempt is otherwise untouched (status/submit_kind stay 'kicked'), so the roster and results reads are unaffected; readmitted_at is moderation metadata of the same class as kicked_by/kick_reason (docs/06:84 immutability protects the answers and the reason, not later moderation bookkeeping). Audited (attempt.readmitted audit_log row per real grant); no realtime event (the student's eventual fresh Start publishes attempt.started, which moves the monitor row). Proven by TestReadmitFlowE2E (auth matrix, not-kicked 409, kicked-row-untouched + one audit row, one-fresh-slot-only, idempotent re-readmit, admin readmit); OpenAPI bumped to v0.11.0 (readmitAttempt) with the web client regenerated. Remaining: per-teacher rate-limiting on kick/readmit (the only rate limiter today is login-only), the student lockout screen + attempt:{id} socket delivery (needs the student WS), guardrail enforcement (violation reporting endpoint, violation_count increments, flag/auto_submit/notify ladder), and the client-side fullscreen/focus/clipboard guardrails.
