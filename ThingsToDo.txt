- Milestone 5 - Realtime (FR-7): server/internal/realtime/ is a doc-comment stub. DONE so far: the GET /quizzes/:id/live roster snapshot (docs/05 section 4) - one row per assigned student collapsed to their latest attempt, owner-or-admin auth via ActionQuizWatchLive, answered_count/question_count/max_score from the pinned snapshot, server_time origin; proven by the TestLiveRosterE2E integration suite and OpenAPI v0.9.0 (getLiveRoster + LiveRosterRow/LiveRosterResponse). Also DONE: the attempt_events source-of-truth layer (docs/05 sections 1-2) - every lifecycle write now appends its typed event row in the same transaction as the state change ("persist first"): attempt.started (fresh start only, resume emits none), attempt.progress (answered_count only; current_question stays null - no server cursor over REST), attempt.submitted for all three kinds (manual funnel + the auto/forced batch sweep, which was rewritten to one transaction with UPDATE ... RETURNING so only rows it actually flipped emit), and attempt.graded (score, once per grade); proven by TestEventFlowE2E. Also DONE: the Redis pub/sub publish brick (docs/05 section 1 "publish second") - realtime.Publisher (go-redis) relays each committed event to quiz:{quiz_id}:events after its transaction commits, injected as the small attempt.EventPublisher interface into the attempt Service (serve-side started/progress/submitted) and threaded into the worker's SweepDueAttempts/GradeSubmitted (worker-side batch submitted + graded), best-effort by contract (a publish failure logs and drops; the live snapshot reconciles), proven by TestEventPublishE2E (capture-fake: right channel/payload, publish-only-after-commit, no publish on rollback/no-flip) plus a Redis-gated TestPublisherLiveSmoke. Also DONE: attempt.progress coalescing (docs/05 section 5) - attempt.ProgressCoalescer, a leading-edge EventPublisher decorator wrapped around the serve-side publisher at the composition root (main.go), relays at most one attempt.progress per 2s per attempt (keyed by attempt_id, one attempt = one student) while every other event type passes straight through, so a 500-student quiz now stays inside the doc's ~250 events/s budget; coalescing is delivery-only (SaveAnswer still appends every progress row, so the append-only log and the reconciling snapshot stay complete), memory is bounded by an amortized sweep-on-write (no janitor goroutine), and the worker is left unwrapped since it emits no progress; proven by the injectable-clock unit test TestProgressCoalescer (rapid burst -> 1 relay, window boundary admits the next, per-attempt independence, non-progress always passes, sweep bounds the map). Remaining: the WebSocket gateway subscribing to those channels, channel subscribe auth (quiz:{id}:monitor / attempt:{id} / user:{id}:notify), heartbeat/disconnect tracking, wiring the events into the live snapshot's current_question, and the teacher live dashboard UI.

Smaller design-doc items with no home yet

- Admin console UI - user/group management exists only as API; the admin sees a placeholder screen.
- Audit log read side - writes happen everywhere, but the admin-facing GET /audit endpoint is missing.
- Live-quiz teacher controls - extend ends_at, force-close early, and archive (the fifth lifecycle state) have no endpoints; only the automatic scheduler close exists.
- Notifications on assignment changes (in-app/email) - nothing exists.
- Single active session per attempt - listed as "always on" in the design; not implemented (depends on the realtime layer).

In short: the server-authoritative core (identity, authoring, scheduling, attempts, grading) is done and solid,but the platform is not yet usable by a student or admin through the browser. The natural next steps in plan order are finishing Milestone 4 (results release + the web attempt player, which is also the exit criterion's user-visible half), then realtime, guardrails/kick, import, and analytics.
