The append-only log is unbounded, so this is the codebase's first paginated read: keyset pagination on the bigserial primary key (ORDER BY id DESC, LIMIT clamped to 1..200 default 50, cursor `before` = id, next_cursor null on the last page - stable under concurrent appends, unlike OFFSET). The read owns its SQL in the audit package (audit.List/Filter/Entry/Page) on the *sql.DB pool (outside any mutation tx); actor_id/resource_id/detail are all nullable columns, so detail scans through a []byte (a direct *json.RawMessage rejects a NULL jsonb and would 500 the whole page) and the nullable ids scan via sql.NullString. Proven by TestAuditReadE2E (admin 200 / teacher+student 403 / unauth 401 gate, filter isolation, a seeded NULL-actor+NULL-detail row that comes back with actor_id omitted and detail=null, from/to range, a keyset walk over 5 rows at limit 2 asserting strictly-descending ids with no cross-page overlap and a null final cursor, and the 422 matrix).
