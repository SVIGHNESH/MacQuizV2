Milestone 4 - The Attempt Player

Everything between "a student clicks Start quiz" and "they read their released score next to the answer key": the transactional attempt lifecycle, server-authoritative timing, deterministic grading, the results-release policy, and the student web UI that finally makes the platform usable from a browser. Implements FR-6 of docs/12-implementation-plan.md.

Exit criterion, met and witnessed: a student takes a quiz, closes the laptop, is auto-submitted at the deadline by the server alone, gets graded - and (the user-visible half) takes another quiz entirely in the browser and reviews the released result with the answer key. Both proven live against the Docker Compose stack.

1. The attempt lifecycle at a glance

FR-6 · docs/08-attempt-lifecycle.md

An attempt is born in one transaction, lives entirely against a pinned snapshot version, and dies exactly once through a single funnel - no matter who kills it:

student clicks Start | v in_progress ---- autosave upserts (PUT answer, write-gated by deadline+status) | | one of four submit kinds, all through the same idempotent funnel: | manual - the student clicks Submit | auto - the attempt_deadline job fires at deadline_at + grace | forced - the quiz closes while the attempt is still open | kicked - (reserved for Milestone 6) v submitted ---- grading job (enqueued transactionally with the submit) | v graded ---- release gate (release_policy auto at close, or the teacher's click) | v score + answer key visible to the student

Two ideas repeat everywhere below. The server owns time: every write validates against a deadline precomputed once, on the database clock; the browser countdown is cosmetic. Every job is an idempotent repair: deadline jobs, sweeps, grading, and release can fire late, twice, or concurrently and converge on the same state.

2. Attempt start: one transaction, one deadline

POST /quizzes/:id/attempts · server/internal/attempt

Starting an attempt is a single transaction that checks the three admission rules and precomputes the one timestamp everything else validates against:

  • Assignment - the caller must hold an assignment row; anyone else reads 404, so quiz existence is never leaked.
  • The live window - judged on server time with the same lazy status derivation readers use (409 QUIZ_NOT_LIVE outside it), so starting the second a quiz goes live never races the scheduler.
  • The attempt budget - attempts used vs max_attempts (409 ATTEMPT_LIMIT_REACHED).

The transaction stamps deadline_at = least(now() + duration_sec, quiz.ends_at) - a student starting late gets the remainder of the window, never time past close. The response is the full player payload: the attempt, the quiz title, the snapshotted guardrail config, the saved answers, the server clock (now) for the client's clock-offset estimate, and the pinned snapshot's questions with the answer key structurally absent - the key lives behind a serializer boundary (json:"-"), so no student-facing payload can carry it by accident.

Restart is idempotent: if an unexpired attempt is already open, the same call returns it with 200 instead of burning an attempt slot - a crashed tab costs nothing. When the quiz shuffles questions, positions are renumbered densely in the per-attempt order, and every later read (including the released review) preserves that same order.

3. Autosave, resume, and the submit funnel

PUT /attempts/:id/answers/:questionId · GET /attempts/:id · POST /attempts/:id/submit

Autosave

Each answer is an idempotent upsert on (attempt_id, question_id); the question must belong to the snapshot version the attempt pinned. Every write passes one gate - the attempt is still in_progress and now() has not passed deadline_at plus a 5-second grace - and is refused with 409 otherwise. The response carries deadline_at and now back, so the client re-estimates its clock offset on every save.

Resume

GET /attempts/:id (owner-only, 404 to anyone else) returns the same player payload as start: saved answers, deadline, server clock. Closing the laptop and reopening it - or a browser crash - loses nothing that was autosaved.

The submit funnel

All terminations go through one submit(attempt_id, kind) path. Whatever is autosaved becomes the submission - there is no separate "final answers" write. Submitting an already-submitted attempt answers 200 with the unchanged terminal state, so a retried request, a second tab, or a race against the deadline job resolves cleanly. The manual leg is the only one a student can reach; auto and forced belong to the worker (next section), and kicked is reserved for the Milestone 6 guardrail ladder.

4. The deadline timer and force-submit on close

River jobs: attempt_deadline, the shared sweep pass · server/internal/worker

"The disappearing student" is handled entirely server-side. Every attempt start transactionally enqueues an attempt_deadline River job scheduled for deadline_at + grace - the job exists if and only if the attempt does. When it fires, it auto-submits whatever is still open (submit_kind: auto).

Behind the exact-timestamp jobs runs an idempotent attempt sweep in the worker's shared pass (also the backstop that paces itself every minute): first the auto leg (any attempt past its deadline), then the forced leg (any attempt still open on a quiz that closed early - submit_kind: forced). A quiz close therefore force-submits its stragglers in the very pass that closes it, and a missed or duplicated job is a harmless repair, never a double submission.

The browser countdown is cosmetic. If every client dies, the server still terminates, grades, and releases every attempt on time - that is the milestone's exit criterion, and the E2E witnesses it with no API call from the student at all.

5. The grading job: deterministic scores from the snapshot key

River job: grade_submitted · server/internal/grading

Grading is a pure function of the submitted answers and the pinned snapshot's answer key - deterministic and idempotent, so re-running it can only rewrite the same result. The manual submit enqueues the grading job in the same transaction; the worker's shared sweep pass also grades, so auto- and force-submitted attempts are graded in the very pass that terminated them.

All four question types score against the key:

TypeResponse shapeCorrect when
singleoption key (string)it equals the key's option
multioption keys (array)the sets match exactly - no partial credit
truefalsebooleanit equals the key
shortstringit normalizes to one of the accepted answers

Each question writes is_correct and points_awarded; unanswered questions keep is_correct null and award zero. The attempt's score is the sum; max_score always derives from the snapshot version the attempt pinned, so a republished quiz never re-scores old attempts.

6. Results release: policy, teacher table, student review

Migration 00004 · quiz.ReleaseDueResults · OpenAPI v0.8.0

When students see scores is a per-quiz decision made at publish: release_policy is auto (the default - the worker releases at close once grading lands) or manual (the teacher clicks). The single fact every student-facing results read gates on is quizzes.results_released_at; while it is null, scores and the answer key stay server-internal.

Auto-release in the shared pass

ReleaseDueResults runs after grading in the worker's shared sweep: a closed auto-policy quiz releases only once no attempt is left in_progress or submitted. One idempotent pass therefore closes the quiz, force-submits stragglers, grades them, and releases - results never appear with half the class ungraded.

The teacher decides with numbers in hand

GET /quizzes/:id/results (owner) lists one row per assigned student per attempt - students who never started keep their attempt fields null, so absence shows instead of hiding - with scores visible as grading lands, before any release: releasing is the teacher's decision, so they need the numbers first. POST /quizzes/:id/release-results stamps the release: refused with 409 QUIZ_NOT_CLOSED until the window ends, idempotent on repeat, audited as quizzes.results_released exactly once.

The student's released review

GET /attempts/:id/result answers 409 RESULTS_NOT_RELEASED until the quiz is released and the attempt graded, then returns the score, the max, and every snapshot question in the player's per-attempt order - the student's response, is_correct, points_awarded, and the correct answer. This is the one documented exception to answer-key isolation, and the only read anywhere that serializes the key.

Release-gated scores in the assigned list

GET /quizzes/assigned carries results_released_at and the caller's own attempt summaries, with the score gated in the SQL itself (CASE WHEN results_released_at IS NOT NULL) - the withheld value structurally never reaches the wire, the same stance as the answer key's serializer boundary.

7. The student web UI: assigned list, player, review

web/src/player/ · typed client over OpenAPI v0.8.0

The student half of the SPA - previously a placeholder screen - is now a real workspace in the same design system (docs/11) as the teacher's:

  • My quizzes (AssignedList): one card per assigned quiz - window, time limit, snapshot question count, the caller's attempt history - offering exactly the one action the state allows: Start quiz, Resume attempt, Review, or a plain explanatory note. Finished attempts read Score withheld until release, then show points and the Review button; the client never computes or guesses a score.
  • The player (AttemptPlayer): renders all four question types with the workspace's option-row language, a countdown pill driven by deadline_at plus a clock-offset estimate refreshed on every save, and per-question debounced autosave (600 ms) with a Saving / All changes saved / Not saved badge. A save landing after a newer edit never clobbers it.
  • Time-up and terminal handling: at zero the player locks and fires the manual submit once - inside the server's grace it lands; past it the deadline job owns the submission and the 409 resolves to the same "Time is up" panel. Any autosave answering 409 (submitted in another tab, quiz force-closed) locks the player instead of arguing with the server.
  • Submit: an inline confirm shows the unanswered count, flushes whatever the debounce still holds, then enters the idempotent funnel.
  • The review (ResultReview): the score banner, per-question verdict chips (Correct / Incorrect / Not answered), every choice row tagged Your answer / Correct answer in success/danger tones, and short answers as a your-answer / accepted-answers pair. The answer key appears exactly here and nowhere else in the client; a 409 renders as "not released yet", never as an error.

9. How it was verified

make test (Go, against Compose Postgres) · node web/e2e/*.e2e.mjs (Chromium)

  • Attempt player backend: integration suite covering the start transaction, idempotent restart, key-free payloads, autosave gates, resume, and every 409 - plus a live Compose E2E over curl.
  • Deadline timer and close: a 6-subtest integration suite and a 12-check live Compose E2E where the worker flipped attempts to submitted/auto and submitted/forced with no API call.
  • Grading: 21 unit cases across the four types, a 5-subtest integration suite, and a 17-check live E2E where the worker graded a manual submit and a disappeared student entirely server-side.
  • Results release: an 11-subtest integration suite (releaseflow_test.go) - policy storage, withheld scores, both 409s, the teacher table pre-release, manual never auto-releasing, idempotent audited release, the released review with key, cross-student 404, and one-pass close→force-submit→grade→release.
  • The web player: a 26-check browser E2E (web/e2e/player.e2e.mjs) - a real student in Chromium starts a live 4-type quiz, autosaves, survives a cold reload via Resume with every answer intact, submits, sees the score withheld, then - after the worker closes, grades, and auto-releases at ends_at - reads the 4/6 review with verdict chips, the missed multi's full key, and the accepted short answer.
All suites green as of 2026-07-07. The browser E2E runs unattended in ~4 minutes: make up, make web-dev, then node web/e2e/player.e2e.mjs.

10. What deliberately remains

  • The Redis snapshot cache for attempt start - listed in the plan as a read-path optimization; Redis is in Compose and config, nothing consumes it yet. The start transaction reads the snapshot from Postgres, which is correct and fast at current scale; the cache is worth adding before the 1,000-student load test (Milestone 9).
  • submit_kind: kicked - the funnel accepts it; the guardrail ladder and the kick endpoint that use it are Milestone 6.
  • Realtime presence - the player writes no attempt_events yet; the WebSocket gateway and the teacher's live roster are Milestone 5.
