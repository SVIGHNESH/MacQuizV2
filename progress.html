Milestone 3 - The scheduler: River jobs flip quizzes live and closed on time

milestone 3
2026-07-06 · after 2cdd57a
  • River is wired in (v0.40, the Postgres-backed job queue the architecture locked in): the worker Compose container is no longer a no-op - it runs a real River client consuming open_quiz, close_quiz, and a periodic sweep_quizzes backstop. River’s own queue tables migrate inside the same macquiz migrate entrypoint (goose for our schema, River’s migrator for its own), so the one-shot Compose migrate service still brings a clean volume all the way to head.
  • Publish now enqueues its transitions transactionally: the open_quiz/close_quiz jobs are inserted in the same transaction as the snapshot and status flip, scheduled at exactly starts_at/ends_at - a scheduled quiz can never exist without its jobs, and a refused publish leaves no orphans (proven in the new integration test). The quiz service carries an insert-only River client over the same *sql.DB; only the worker process ever works jobs.
  • Every trigger funnels into one idempotent sweep: SweepDueQuizzes closes scheduled/live quizzes past ends_at, then opens scheduled quizzes past starts_at, purely from the row’s own window. Jobs carry only the quiz id, so a job firing for a rescheduled quiz, firing twice, or firing a day late is a harmless no-op or a repair - never a wrong flip. A quiz whose whole window elapsed while the worker was down goes straight to closed, never momentarily live.
  • The docs/02 section 4.6 backstop is real: the worker re-scans for due-but-unfired transitions at boot (an outage can never leave a quiz stuck) and repeats the sweep every minute as a periodic River job, while readers keep the lazy status derivation from the last entry - three independent layers agreeing on one state machine.
  • Republish stays safe: rescheduling writes a fresh job pair for the new window; the stale pair stays queued and no-ops against the sweep predicates when it fires - pinned by the integration test alongside the straight-to-closed path and sweep idempotency.
  • Tested at the process boundary too: a new worker integration test (own database, macquiz_workertest) publishes a quiz with a seconds-long window through the real service, runs the real worker.Run, and watches the stored row - not a derived read - flip scheduled → live → closed, then shuts the worker down cleanly. The migration lifecycle test now exempts River’s self-versioned river_* tables from the goose down-to-zero check.
  • Force-submit on close is deliberately absent: no attempts exist until Milestone 4; the submit funnel will hang off the same closed transition when it lands.
Milestone 3 exit criterion verified live against the Compose stack from clean volumes: a provisioned teacher published a quiz with a 40-second-away window over curl, and the stored row flipped to live one second after starts_at and to closed at ends_at exactly, with no request touching the API - worker logs show both flips fired from their exact-timestamp jobs. The Milestone 3 backend is complete; the remaining item is the web publish/assign UI.

Milestone 3 - Publish with immutable snapshots, assignments, and the student quiz list

milestone 3
2026-07-05 · after a543d20
  • Publishing is now real: POST /api/v1/quizzes/:id/publish takes the live window, the per-attempt duration, and the guardrail config, and transitions Draft to Scheduled in one transaction. The docs/04 preconditions are enforced as field errors - at least one question, starts_at < ends_at and in the future, a sane duration (30 s to 24 h), and at least one assigned student.
  • Immutable version snapshots (new quiz_versions table, migration 00003): publish copies the full question rows - including the answer key - plus the guardrails into a write-once version row, so a mid-window edit or republish can never change what a student who already started sees. The copy happens in SQL on purpose: the Go Question type structurally drops the answer key when serialized, which is exactly right for clients and exactly wrong for the grader. The table rejects UPDATE and DELETE by trigger; republishing while scheduled reschedules and writes version n+1.
  • Assignments with group expansion: PUT /quizzes/:id/assignments replaces the audience atomically from direct student ids and/or group ids. Groups are expanded to individual rows at assignment time, so removing a student from a group later never silently revokes an already-assigned quiz - proven in the integration test by emptying the group and watching the audience stay intact. Non-student ids or unknown groups change nothing; a live quiz's audience is frozen (409); GET /quizzes/:id/assignments lists the current audience for the editor UI.
  • The student flow begins: GET /api/v1/quizzes/assigned lists the caller's upcoming, live, and past quizzes - visibility is the assignment row itself, drafts and archived quizzes never appear, and the question count comes from the published snapshot. The response shape carries no owner, no guardrail internals, and structurally no questions.
  • Lazy state validation: readers derive the effective status from the window, so a scheduled quiz whose starts_at has passed reads live (and past ends_at reads closed) on both the student list and the teacher's own screens, before any scheduler job flips the row - the docs/06 rule that no reader ever sees a stale state. The River open_quiz/close_quiz jobs that flip the stored row (and force-submit attempts at close) are the next unit.
  • Guardrails vocabulary pinned: fullscreen and focus tracking as off/warn/count, clipboard blocking, and the violation ladder (1-100 counted violations, then flag / auto_submit / notify) validate at the boundary and default to the documented flag-at-3; the config is snapshotted with the questions.
  • Audited and contract-first as always: quizzes.published (with version and window) and quizzes.assignments_set rows commit with their mutations; api/openapi.yaml is at v0.5.0 with the four new operations and the Guardrails/AssignedQuiz schemas, regenerated into the typed web client (drift check green).
  • Tested at three levels: unit tests pin the status derivation table and the guardrail vocabulary; a new integration test in its own database (macquiz_lifecycletest) drives audience-first publishing, group expansion, window rule breakers, snapshot immutability (the UPDATE is rejected by Postgres), republish version bumps, student visibility vs. the unassigned outsider, the lazy live transition, and audit counts.
Verified live against the Compose stack over curl: a provisioned teacher authored a quiz, was refused publish until a student was assigned, published with guardrails, and the snapshot row held the answer key; the assigned student's GET /quizzes/assigned showed the quiz as scheduled, then as live the moment starts_at passed - with no scheduler running at all. The scheduler jobs and the web publish/assign UI are the remaining Milestone 3 items.

milestone 2

  • Question CRUD with per-type validation (POST /quizzes/:id/questions, PATCH/DELETE /questions/:id): the docs/07 rules are enforced as field errors - a correct answer must exist among the options, points > 0, choice types need 2-8 options with unique keys, true/false takes a boolean, and short answers need at least one accepted string.   • Reorder with dense positions: PUT /quizzes/:id/questions/order accepts only a full permutation of the quiz's question ids (anything else changes nothing) and rewrites positions 1..n in one statement; deleting a question re-densifies the survivors in the same transaction, so position is always a clean dense integer.
  • The serializer boundary, tested from day one: the internal Question type tags the answer key json:"-", so the default serialization structurally cannot leak it; handlers must opt in via the owner-facing TeacherQuestion view, and the student view drops the key from the value itself.   • Authorization through the central policy: the authoring surface is teacher-only (admin and student get 403 - admins cannot author, per the security doc), and every per-quiz operation checks ownership via Can, answering 404 - never 403 - to non-owners so quiz existence is not leaked. Every mutation writes its audit row (quizzes.created/updated/deleted, questions.created/updated/deleted/reordered) in the same transaction, via a new shared audit package.
  • Contract updated to v0.4.0: nine new operations plus the Quiz, Question, TeacherQuestion, and QuestionInput schemas in api/openapi.yaml, regenerated into the typed web client (drift check green).
  • Tested at three levels: table-driven validation tests, the serializer-invariant tests, and an HTTP-level integration test in its own database driving the full authoring flow - creation, all four types, rule breakers, cross-teacher 404s, reorder, delete re-densify, the published-quiz gate, cascade delete, and exact audit-row counts.

Milestone 1 - Authentication core: login, sessions, forced reset

milestone 1
2026-07-05 · after 34663f4
  • Argon2id password hashing in PHC string format (OWASP parameters recorded per hash, so they can be raised later without invalidating old hashes), with a constant-time decoy verification so unknown-email and wrong-password logins cost the same.
  • Sessions per the security doc: 15-minute HS256 JWT access tokens plus opaque rotating refresh tokens, both in httpOnly cookies; the refresh cookie is path-scoped to /api/v1/auth. Only the SHA-256 of a refresh token touches the database (new user_sessions table, migration 00002). Replaying a used or revoked token is treated as theft and revokes the whole session family.
  • Auth endpoints live: POST /api/v1/auth/login (rate-limited per IP and per account, 429 + Retry-After), refresh, logout, GET me, and POST password - all added to api/openapi.yaml with the shared error vocabulary and regenerated into the typed web client.
  • Forced first-login reset: accounts start with must_change_password = true; the RequirePasswordChanged middleware answers 403 PASSWORD_CHANGE_REQUIRED on every module route until the user sets their own password, which also revokes all sessions and writes the audit_log row in the same transaction.
  • Admin bootstrap: new idempotent macquiz bootstrap mode creates the self-referencing first admin (audited); Compose runs it as a one-shot service after migrate, before app.
  • Readiness probe: GET /readyz now answers 503 until Postgres does, complementing the dependency-free /healthz.
  • Tested at both levels: unit tests for hashing, JWT, and the rate limiter; an HTTP-level integration test in its own database covering bootstrap idempotency, login, refresh rotation, reuse detection, the forced-reset gate, password change, logout, and the per-account login limit.
Verified live: from clean volumes the stack migrates, bootstraps the admin, and serves; curl login sets both cookies, /me and refresh rotation work, and refresh after logout is rejected. All Go and web checks green.
