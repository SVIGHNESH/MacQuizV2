Milestone 5 - The live roster snapshot: the dashboard's first brick

milestone 5
2026-07-08 · after b520fa8
  • The snapshot the dashboard opens on (GET /quizzes/:id/live, docs/05 section 4): before the WebSocket layer exists, the teacher can already read the current roster over plain REST - the same materialized state the live dashboard will fetch on connect and then apply streamed deltas onto. One row per assigned student, collapsed to their latest attempt via a LATERAL join to max(attempt_no), so a student who restarted is one cell, not two - the docs/05 section 6 rule that a student is in exactly one roster state.
  • State is derived, honestly degraded: state folds the attempt status into the dashboard vocabulary - not_started (assigned, never started), in_progress, submitted (covers graded), kicked.   • Authorized as the owning teacher or any admin (docs/05 section 3): the service gates on the existing ActionQuizWatchLive policy action, so an admin can open the dashboard for a quiz they do not own while a non-owning teacher reads 404 (existence never leaks); a new requireStaff route group answers 403 to students before the handler.
  • Proven by a new integration suite (liveflow_test.go, own database macquiz_livetest): three assigned students in distinct states (one in progress with one saved answer, one submitted, one never started), a directly-inserted stale earlier attempt to prove the roster collapses to the latest and still emits exactly one row per student, the question_count/max_score from the snapshot, an admin watching a quiz it does not own, the non-owning teacher's 404, and the student's 403 - the full quiz and attempt suites still green beside it. Milestone 4 - The grading job: deterministic scores from the snapshot key

milestone 4
2026-07-06 · after 1d87b43
  • Grading is a pure function of (snapshot, answers) - docs/03 invariant 4, verbatim. The grader (attempt/grade.go) locks a submitted attempt, reads the answer key from the quiz_versions row the attempt pinned - never the live quiz, so a republish after submission cannot change what work is graded against - writes is_correct and points_awarded per answer, sums the score, and flips the row to graded behind the funnel's own status = 'submitted' guard, so a job firing twice, late, or concurrently with the backstop can never double-grade.
  • The docs/04 rules per type: choice questions compare keys (multi is all-or-nothing set equality - no silent partial credit), truefalse compares booleans, and short answers use normalized exact/numeric matching: case-insensitive, trimmed, inner whitespace collapsed, and "5.0" matches an accepted "5" whether the widget saved a string or a bare number. Autosave accepts opaque JSON, so the grader is where malformed input lands: a response that does not decode into its type's shape grades as wrong - it never errors an entire attempt.
  • Two trigger paths, one funnel: the manual submit leg enqueues a grade_attempt job inside the submitting transaction (docs/04: "submission enqueues a grading job"; a repeat submit enqueues nothing), while the batch legs need no job at all - the worker's shared sweep pass now runs quiz flips, attempt terminations, then GradeSubmitted, so an auto- or force-submitted attempt is graded in the very pass that terminated it, and the boot re-scan plus minutely backstop heal any grading ever missed (the live run's boot re-scan promptly graded two attempts left submitted by the previous milestone's E2E).
  • The score stays withheld: grading writes the column, but the player payload still carries no score and no answer key - results become visible when the release policy lands (docs/01: auto at close or explicit teacher release). No OpenAPI change: graded was already in the status enum, so the whole feature is contract-neutral.
  • Proven at three boundaries: table-driven unit tests pin every per-type rule including the normalization and garbage cases; a new integration suite (gradeflow_test.go, own database) proves the grade job commits with the manual flip and only once, the 2+0+1+2 score across all four types with per-answer marks, regrade-is-a-no-op, the sweep pass grading the disappearing student it auto-submits with zero jobs, and the score/key withheld from the player; and a 17-check live Compose E2E where the worker graded a manual submit within seconds (graded/5) and a vanished student's attempt went graded/auto/1 entirely server-side - both flips witnessed in worker logs.
The Milestone 4 exit criterion now runs end to end: a student took a quiz, closed the laptop, was auto-submitted at the deadline, and got graded - by the server alone, against the Compose stack. Milestone 4 - The disappearing student: deadline timer and force-submit on close

milestone 4
2026-07-06 · after 69cb199
  • Every attempt now carries its own alarm clock: the start transaction enqueues an attempt_deadline River job scheduled at deadline_at + 5s grace, committing with the attempt row itself - an attempt is never created without its auto-submit, a failed start leaves no orphan job, and an idempotent resume enqueues nothing (one timer per attempt, pinned by test). The job fires after the grace window on purpose: the write gate already refuses everything past that moment, so the last in-flight autosave lands before the terminal flip.
  • The attempt sweep mirrors the quiz sweep (attempt.SweepDueAttempts): two idempotent statements behind the funnel's own status = 'in_progress' guard, so the sweep and a racing manual submit can never double-terminate. First auto - attempts past their personal deadline_at plus grace - then forced - attempts still open on a closed quiz (docs/06: "close_quiz force-submits all open attempts"). Auto runs first so an attempt that expired before its quiz closed keeps the kind its own timer would have written; the autosaved answers survive untouched for the grader.
  • One shared sweep pass in the worker: the deadline job, the close_quiz transition, the boot re-scan, and the minutely backstop all run quiz flips then attempt terminations in the same pass - a quiz closed in a pass has its open attempts force-submitted in that pass, and a job firing late or twice repairs everything due, never wrong-flips. The jobs still carry nothing but an id; state is always re-derived from the rows.
  • No contract change: submit_kind already enumerated auto and forced in OpenAPI v0.7.0, and no new endpoints exist - the whole feature is server-side time authority.
  • Proven at two boundaries: a new integration suite (deadlineflow_test.go, own database) pins the timer job's exact scheduled_at, resume-enqueues-nothing, the no-op sweep inside the budget, auto-submit past deadline+grace with the neighbor attempt untouched, force-submit on close with the auto kind preserved, and terminal idempotence - plus a 12-check live Compose E2E where a student autosaved an answer and "closed the laptop": the worker's attempt_deadline job flipped the row to submitted/auto with no API call, a second attempt was flipped to submitted/forced by the close_quiz job at ends_at, the stale tab's next autosave read ATTEMPT_ALREADY_SUBMITTED, and the saved answer survived for the grader.

Milestone 4 - The attempt player backend: start, autosave, resume, submit

milestone 4
2026-07-06 · after a4aa9cb
  • The start transaction is one atomic decision (POST /quizzes/:id/attempts): the assignment row doubles as the visibility check (unassigned students read 404, so quiz existence never leaks) and, locked FOR UPDATE, as the serialization point - concurrent starts by the same student queue up instead of racing attempt_no. Inside the same transaction: the live-window check on the database clock (answering QUIZ_NOT_LIVE before starts_at and after ends_at), the max_attempts budget (ATTEMPT_LIMIT_REACHED), and the deadline precompute deadline_at = least(now() + duration, ends_at) - computed once, the single column every later write validates against.
  • Starting is idempotent: an unexpired in-progress attempt is returned (200) instead of duplicated, so a reloaded tab or a second device resumes rather than burning an attempt slot - the row count stays at one, pinned by test.
  • The serializer boundary extends to the player: attempts pin quiz_version and serve questions from the immutable snapshot, decoded into a type whose answer key is tagged json:"-" - carried internally for the coming grader, structurally absent from every player payload. Shuffled quizzes get a deterministic per-attempt order (hash of attempt id + question id, positions re-densified), so every resume sees the same order with no order storage.
  • Autosave is an idempotent upsert with one write gate (PUT /attempts/:id/answers/:qid): one row per question, last write wins, and every write passes the same checks - owner (404), in-progress status (ATTEMPT_ALREADY_SUBMITTED / ATTEMPT_KICKED), and the docs/06 deadline-plus-5s-grace judged on the database clock that set the deadline (ATTEMPT_DEADLINE_PASSED). The question must exist in the attempt's own snapshot version, so a mid-window republish can never make a stale player write outside what its student saw. Responses are size-capped; resume (GET /attempts/:id) returns the saved answers, the deadline, and the server clock for the cosmetic countdown.
  • The submit funnel exists and manual rides it (POST /attempts/:id/submit): one submit(attempt, kind) routine holds the terminal-state rules - kicked refuses, submitted/graded returns unchanged (a double-click or a race against the future deadline job resolves to whichever committed first), and only the student-driven manual kind races the clock. Auto, forced, and kicked will enter through the same door; grading enqueue joins the funnel when the grading job lands.
  • Proven at two boundaries: a new integration suite (playerflow_test.go, own database, eleven subtests over real HTTP) covers the early-start refusal, the outsider 404 and teacher 403, deadline precompute, key stripping, idempotent restart, upsert-not-duplicate autosave, garbage rejection, resume, the submit funnel, the expired-attempt write gates, limit exhaustion, and the kicked lockout codes - plus a 16-check live curl E2E against the Compose stack from clean volumes where the open_quiz job flipped the quiz and a student played an attempt end to end. OpenAPI is at v0.7.0 with the four player operations, regenerated into the typed client.

Milestone 3 - The scheduler: River jobs flip quizzes live and closed on time

milestone 3
2026-07-06 · after 2cdd57a
  • River is wired in (v0.40, the Postgres-backed job queue the architecture locked in): the worker Compose container is no longer a no-op - it runs a real River client consuming open_quiz, close_quiz, and a periodic sweep_quizzes backstop. River’s own queue tables migrate inside the same macquiz migrate entrypoint (goose for our schema, River’s migrator for its own), so the one-shot Compose migrate service still brings a clean volume all the way to head.
  • Publish now enqueues its transitions transactionally: the open_quiz/close_quiz jobs are inserted in the same transaction as the snapshot and status flip, scheduled at exactly starts_at/ends_at - a scheduled quiz can never exist without its jobs, and a refused publish leaves no orphans (proven in the new integration test). The quiz service carries an insert-only River client over the same *sql.DB; only the worker process ever works jobs.
  • Every trigger funnels into one idempotent sweep: SweepDueQuizzes closes scheduled/live quizzes past ends_at, then opens scheduled quizzes past starts_at, purely from the row’s own window. Jobs carry only the quiz id, so a job firing for a rescheduled quiz, firing twice, or firing a day late is a harmless no-op or a repair - never a wrong flip. A quiz whose whole window elapsed while the worker was down goes straight to closed, never momentarily live.
  • The docs/02 section 4.6 backstop is real: the worker re-scans for due-but-unfired transitions at boot (an outage can never leave a quiz stuck) and repeats the sweep every minute as a periodic River job, while readers keep the lazy status derivation from the last entry - three independent layers agreeing on one state machine.
  • Republish stays safe: rescheduling writes a fresh job pair for the new window; the stale pair stays queued and no-ops against the sweep predicates when it fires - pinned by the integration test alongside the straight-to-closed path and sweep idempotency.

The migration lifecycle test now exempts River’s self-versioned river_* tables from the goose down-to-zero check.
  • Force-submit on close is deliberately absent: no attempts exist until Milestone 4; the submit funnel will hang off the same closed transition when it lands.
