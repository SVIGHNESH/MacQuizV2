Version 2 adds in-attempt integrity guardrails and live moderation: the teacher can now remove a candidate from a quiz mid-attempt.

Section 01

Overview & goals

MacQuiz is a closed-enrollment assessment platform. There is no self-signup: a single admin (or admin team) creates every teacher and student account. Teachers own the full quiz lifecycle - authoring, scheduling, audience selection, live monitoring, and post-quiz analytics - while the admin has organization-wide visibility over both teachers and students.

Primary goals

  • Strict access control. Every capability is derived from role plus ownership. A teacher sees only their own quizzes and assigned students; a student sees only quizzes assigned to them, and only inside the live window.
  • Server-authoritative timing. The quiz opens, closes, and auto-submits based on server time. Client clocks are never trusted.
  • Real-time monitoring. While a quiz is live, the owning teacher watches attempts progress question-by-question with sub-second latency.
  • Layered analytics. Teachers analyze their students and quizzes; the admin analyzes everything, including teacher activity itself.
  • Attempt integrity (v2). Browser-level guardrails (fullscreen lock, focus tracking, clipboard blocking) feed a violation ladder, and the owning teacher can remove a candidate from a live attempt. Every signal and action is server-recorded and auditable.

Non-goals (v2)

  • Camera-based proctoring (webcam / screen recording). v2 covers browser-level guardrails (section 10); camera and screen capture remain out of scope, though the violation event stream leaves room for them.
  • Subjective grading workflows (essay questions with manual marking). v1 supports auto-gradable types: single choice, multiple choice, true/false, and short numeric/text answers.
  • Public quiz sharing or anonymous access.

Section 03

Roles & permissions

Three roles, one hierarchy. The admin provisions people; teachers provision content and audiences; students consume. Ownership matters as much as role: teacher permissions on quizzes and analytics apply only to quizzes they created and students assigned to those quizzes.

Admin

  • Creates and deactivates all teacher and student accounts
  • Resets credentials, assigns students to class groups
  • Sees analytics across every teacher and student
  • Cannot author quizzes (keeps the audit story clean)

Teacher

  • Authors quizzes: one-by-one or bulk upload
  • Schedules the live window and picks the audience
  • Live-monitors attempts on their own quizzes
  • Sets guardrails and can kick a candidate mid-attempt
  • Analytics scoped to their own quizzes and students

Student

  • Sees only quizzes assigned to them
  • Can start an attempt only inside the live window
  • One attempt per quiz (configurable per quiz)
  • Sees own results and history after quiz close

Design decision

Permissions are enforced at the API layer with a single policy function can(actor, action, resource) evaluated on every request, never in the UI alone. The UI merely hides what the API would reject.

Section 04

High-level architecture

A modular monolith behind one API gateway, with a dedicated realtime gateway for WebSockets. The modules (auth, users, quizzes, attempts, analytics) are separate packages with clean interfaces, so any of them can be split into a service later without a rewrite. Redis carries ephemeral state (sessions, live presence, pub/sub); PostgreSQL is the single source of truth.

Why a modular monolith

At this product's scale (thousands of concurrent students, not millions), one well-factored deployable is simpler, more robust, and easier to reason about than microservices. The two pieces that genuinely benefit from independence are pulled out from day one: the realtime gateway (different scaling profile - many long-lived connections, little CPU) and the import/grading workers (bursty, queue-driven). Everything communicates through Redis pub/sub and the job queue, so extraction later is a deployment change, not a design change.

Section 05

Notes on the model

  • Snapshot on publish. Publishing copies the question set into an immutable version. Editing a scheduled quiz creates version n+1; attempts record which version they ran against, so results always match what the student actually saw.
  • deadline_at is precomputed at attempt start as min(started_at + duration_sec, quiz.ends_at). Every autosave and the final submit are validated against this one column - a single comparison, no clock math scattered around the codebase.
  • attempt_events doubles as replay. The realtime dashboard is driven by these events; because they persist, a teacher who opens the dashboard late (or the admin, after the fact) gets an accurate timeline, not just whatever they happened to be connected for.
  • Assignments are rows, not JSON. The student's quiz list is a simple indexed join: quiz_assignments WHERE student_id = ?. Group assignment is expanded to individual rows at assignment time, so removing a student from a group never silently revokes an already-assigned quiz.
  • Kicked attempts are terminal, not deleted. A kick flips status to kicked in the same transaction that records who and why. The row, its answers, and its event trail survive for audit and dispute resolution; only new writes are refused.

Section 06

Quiz lifecycle

A quiz moves through five states. Transitions into and out of live are executed by the scheduler worker on server time, never by a client request.

Draft
Authoring: add, edit, import questions
publish + schedule
Scheduled
Snapshot frozen, audience assigned, visible as "upcoming"
starts_at reached
Live
Students can start attempts; teacher dashboard streaming
ends_at reached
Closed
Open attempts force-submitted, grading runs, results release
teacher archives
Archived
Read-only; analytics retained
  • Draft → Scheduled requires: at least one question, a valid window (starts_at < ends_at, in the future), a duration, and at least one assigned student. The publish action snapshots questions and writes the version.
  • Scheduled → Live / Live → Closed are fired by a scheduler that enqueues an open_quiz / close_quiz job at the exact timestamps. The API additionally validates state lazily on read (if a request arrives after starts_at but before the job ran, the quiz is treated as live), so a delayed job can never block students.
  • Reschedule / cancel is allowed while Scheduled. Once Live, the teacher can only extend ends_at, force-close early, or kick individual students (section 10); all three actions are audited and broadcast to connected students.

Section 07

Question authoring & bulk upload

One-by-one authoring

The authoring UI is a straightforward CRUD flow against POST /quizzes/:id/questions. Each question is validated on save (a correct answer must exist among the options, points > 0, choice types need 2-8 options). Questions are reorderable; position is a dense integer the API rewrites on reorder. Autosave in the editor means a teacher never loses a half-written question.

Bulk upload pipeline

Bulk upload is asynchronous and transactional: the file is fully validated before a single question row is written, and the commit is all-or-nothing so a quiz can never end up with half an import.

  1. Template. The teacher downloads a CSV/XLSX template: type, question, option_a..option_f, correct, points. One row per question.
  2. Upload. The file goes to object storage via a pre-signed URL; the API creates an imports row in state validating and enqueues a job. Limit: 10 MB / 500 rows.
  3. Validate. The import worker parses every row and collects errors per row and column (unknown type, missing correct answer, correct answer not among options, duplicate question text within the file, malformed points). Nothing is written to questions yet.
  4. Review. The teacher sees a preview: n rows parsed, m errors, with each failing row shown inline. If there are errors, the state is failed with a downloadable error report; the teacher fixes the file and re-uploads.
  5. Commit. On a clean file the state becomes ready; the teacher confirms and the worker inserts all rows in one transaction, tagged source = 'import' with the import_id for provenance. Imported questions are ordinary questions afterwards - editable one-by-one like any other.
Why async

Parsing and validating 500 rows with images is too slow for a request cycle, and a synchronous import that dies halfway leaves a corrupted quiz. The job-queue pipeline gives retries, progress reporting, and a clean failure mode for free.

Section 08

Scheduling & audience visibility

The live window

A quiz has three time controls, all set by the teacher at publish:

FieldMeaningEnforced where
starts_atThe earliest moment any assigned student can start an attempt. Before this, the quiz shows as "upcoming" with a countdown.POST /attempts rejects earlier starts; scheduler flips status
ends_atHard close. No new attempts may start, and every open attempt is force-submitted at this moment regardless of remaining personal time.Scheduler job + deadline_at clamp
duration_secPer-attempt time budget, e.g. 30 minutes inside a 2-hour window. A student starting at any point gets min(duration, time-to-close).Precomputed into attempts.deadline_at

All timestamps are stored in UTC and rendered in each user's timezone. The student client shows a countdown driven by the server-provided deadline plus a clock-offset estimate; the client countdown is cosmetic - the server enforces the real one.

Audience selection

Visibility is a whitelist. When publishing (or any time before close), the teacher picks the audience from students the admin has placed in the system: individually via search, or by selecting a group, which expands to individual quiz_assignments rows. A student's quiz list is therefore exactly one query, and there is no path by which an unassigned student can even learn the quiz exists - quiz detail endpoints return 404 (not 403) for unassigned students to avoid leaking existence.

  • Adding students while a quiz is live is allowed (late invite); removing a student who has an in-progress attempt from the assignment is blocked. Interrupting a live attempt goes through the explicit kick control (section 10) instead, so every interruption is attributed, reasoned, and audited.
  • Assignment changes emit notifications (in-app, plus optional email) so students see new quizzes without polling.

Section 09

Attempt flow & timing enforcement

Starting an attempt

POST /quizzes/:id/attempts runs one transactional check: the caller is assigned, the quiz is live (window check on server time), and attempts used < max_attempts. On success it creates the attempt with started_at = now() and deadline_at = least(now() + duration, ends_at), then returns the question set without the correct field - answer keys never leave the server until results are released.

During the attempt

  • Autosave. Every answer change is debounced (2 s) and PUT to /attempts/:id/answers/:questionId. The server rejects writes where now() > deadline_at + grace (grace = 5 s for network jitter). Saves are idempotent upserts keyed on (attempt_id, question_id).
  • Resume. On reload or reconnect, GET /attempts/:id returns saved answers, current server time, and the deadline, so the student lands exactly where they were with an accurate countdown.
  • Heartbeat. The attempt WebSocket sends a heartbeat every 10 s; missing heartbeats mark the student "disconnected" on the teacher's dashboard but do not pause the clock.

Ending an attempt

Three exits, all funneled through the same idempotent submit(attempt_id, kind) routine:

KindTriggerBehavior
manualStudent clicks SubmitValidated against deadline, confirmed with an "n unanswered" prompt
autodeadline_at reachedA per-attempt timer job fires at the deadline; whatever is autosaved becomes the submission
forcedQuiz ends_at / teacher force-closeThe close job sweeps all in-progress attempts of the quiz

Submission enqueues a grading job: choice and true/false questions are compared against the snapshot key; short answers use normalized exact/numeric matching. Grading writes is_correct, points_awarded, and the attempt score, then emits a graded event. Because grading is a pure function of (snapshot, answers), reruns are safe.

Edge case: the disappearing student

A student who closes their laptop mid-attempt never sends a submit. The deadline timer job is the safety net: it fires server-side at deadline_at and auto-submits the autosaved answers. This is why answers are persisted on every change rather than at submit time.

Section 10

Attempt guardrails & kick control v2

v2 adds a light integrity layer to the attempt player, plus the one moderation power teachers asked for most: removing a candidate from a quiz while it is running. The design principle is that client signals are evidence, server actions are enforcement - nothing the browser reports is trusted to block anything by itself, and nothing the browser suppresses can keep the server from terminating an attempt.

Guardrails during an attempt

Guardrails are configured per quiz before publish and snapshotted with the question set, so the rules cannot change under a student mid-window. Each one can be off, warn-only, or counted toward the violation ladder.

GuardrailWhat it doesPolicy options
Fullscreen lockThe attempt player requests fullscreen at start; leaving fullscreen raises a violation and overlays a "return to fullscreen" blocker.off / warn / count
Focus trackingWindow blur and tab switches are reported with their duration (visibilitychange + blur), so the teacher sees "left the tab for 40 s", not just a flag.off / warn / count
Clipboard & context menu blockCopy, cut, paste, and right-click are disabled inside the player; attempts to use them are logged.off / on (logged)
Single active sessionCarried over from v1: starting the attempt from a second device invalidates the first socket and is logged as an event.always on
Violation ladderAfter max_violations counted violations, the configured action fires: flag (dashboard only), auto_submit (attempt submitted as-is), or notify (prompt the teacher to decide). Default: flag at 3.flag / auto_submit / notify

Every violation follows the same pipeline as any attempt event: the client reports it over the attempt WebSocket (REST fallback), the Attempt module increments attempts.violation_count, appends an attempt_events row, and publishes attempt.violation to the quiz channel. The student sees a warning toast ("Violation 2 of 3 - stay in the quiz window"); the teacher's roster row gains an amber badge with the running count and the violation types on hover.

Honest limits

Browser guardrails deter and document; they do not make cheating impossible. A student can use a second device, and a hostile client can fail to report events. That is why violations are framed as evidence for the teacher rather than automatic punishment, why the default ladder action is flag rather than auto_submit, and why the kick decision stays with a human.

Kicking a candidate

The kick is the teacher's server-authoritative remedy, available on every roster row of the live dashboard for their own quizzes (and to the admin on any quiz). It is deliberately a two-step UI - pick a reason (canned list plus free text), then confirm - because it is destructive to the student's session and lands in the audit log verbatim.

Teacher dashboard->POST /attempts/:id/kick {reason} - policy check: caller owns the quiz or is admin
Attempt module->One transaction: status='kicked', submit_kind='kicked', kicked_by, kick_reason, submitted_at=now(); audit row written
Attempt module->Publishes attempt.kicked; enqueues grading of whatever was autosaved
Realtime gateway->Delivers a lockout message to the student's attempt:{id} socket, then force-closes it; fans the event to the monitor channel
Student client->Renders the removed-from-quiz screen with the reason. If the socket message was lost, the next autosave returns 409 ATTEMPT_KICKED and triggers the same screen
  • Enforcement is the status flip, not the socket. The moment the transaction commits, every autosave and submit for that attempt is rejected by the same status = 'in_progress' check that guards the deadline. A student who blocks WebSockets, freezes the tab, or replays requests gains nothing.
  • Kicked work is graded, not discarded. Autosaved answers are graded normally and the result is flagged kicked wherever scores appear; the teacher can override the score to zero per attempt. Discarding work silently would make disputes unresolvable.
  • Re-admission is a new attempt, not a resurrection. If the kick was a mistake, POST /attempts/:id/readmit grants that student one extra attempt slot (audited); they start fresh with whatever time remains in the window. The kicked attempt stays in the record.
  • No kick storms. Kick and re-admit are rate-limited per teacher and always require the reason field - a moderation action you cannot explain is one you should not be taking.
Design decision

Kick reuses the existing idempotent submit(attempt_id, kind) routine with kind='kicked' rather than introducing a parallel termination path. One funnel means the deadline check, grading trigger, event emission, and idempotency guarantees are shared - a kick that races an auto-submit resolves cleanly to whichever committed first.

Section 11

Live attempt tracking

While a quiz is live, the owning teacher's dashboard shows a roster of assigned students, each in one of the states not started / in progress / disconnected / submitted / kicked, with the current question number, answered count, violation badge, and elapsed time updating in real time. The kick control lives on each in-progress row. The admin can open the same dashboard for any quiz (read-only except for the kick escalation power).

Event pipeline

Student client->Action (start, answer autosave, navigate, submit, heartbeat) hits the Attempt module over REST/WS
Attempt module->Persists the change, appends a row to attempt_events, publishes to Redis channel quiz:{quiz_id}:events
Realtime gateway->Subscribed to the channel; fans the event out to every authorized WebSocket on quiz:{quiz_id}:monitor
Teacher dashboard->Applies the event to its in-memory roster; UI cell updates in < 1 s end to end

Event vocabulary

EventPayloadDashboard effect
attempt.startedstudent_id, attempt_id, deadline_atRow moves to "in progress", countdown starts
attempt.progresscurrent_question, answered_countProgress bar and question indicator update
attempt.disconnectedlast_seen_atRow flagged amber "disconnected" (clock keeps running)
attempt.reconnected-Flag cleared
attempt.violationtype, duration_ms, violation_countAmber badge increments; violation type shown on hover (v2)
attempt.kickedkicked_by, reasonRow moves to "kicked"; student client shows lockout screen (v2)
attempt.submittedsubmit_kind, answered_countRow moves to "submitted"; summary counters update
attempt.gradedscoreScore appears (visible per quiz result policy)
quiz.extended / quiz.closednew ends_atBanner to teacher and all in-progress students

Design details

  • Authorization at subscribe time. Joining quiz:{id}:monitor requires being the quiz owner or an admin; joining attempt:{id} requires owning that attempt. The gateway checks once at subscribe and revalidates on token refresh.
  • Snapshot + delta. On connect the dashboard first fetches GET /quizzes/:id/live (current roster state materialized from attempts + recent attempt_events), then applies streamed deltas. This makes late joins and reconnects consistent - no missed-event drift.
  • Throttling. attempt.progress is coalesced per student to at most 1 event/2 s; a 500-student quiz peaks around 250 events/s, trivial for Redis pub/sub and a single gateway node.
  • Degradation. If the WebSocket cannot connect (restrictive school network), the dashboard falls back to polling the snapshot endpoint every 10 s. Students' attempts never depend on the socket - REST autosave is the primary write path.

Section 12

Analytics

Analytics answer three questions at three altitudes: how did a student do, how did a quiz perform, and how active/effective is a teacher. The first two are visible to the owning teacher and the admin; the third is admin-only (teachers see their own).

What is computed

LevelMetricsVisible to
Per student Score and percentile per quiz, accuracy trend over time, average time per question, completion rate (assigned vs attempted), strength/weakness by topic tag Owning teacher, admin, the student (self)
Per quiz Score distribution (histogram), mean/median, participation rate, per-question item analysis: difficulty (p-value), discrimination (point-biserial), option-pick rates, average time on question, integrity summary (violations per student, kicked attempts) Owning teacher, admin
Per teacher Quizzes created/conducted, total student attempts, average participation, average class score, grading turnaround (auto, so effectively publish-to-results latency) Admin (teacher sees own)
Org-wide Active users, quizzes per week, platform participation, cohort comparisons across groups Admin

How it is computed

v1 needs no separate analytics store. Two mechanisms cover everything:

  • Rollup on close. When a quiz closes and grading finishes, a job writes one quiz_stats row (distribution buckets, mean, item-analysis JSON) and upserts student_stats. Dashboards read these precomputed rows, so the heavy queries run once, not per page view.
  • Read replica for exploration. Ad-hoc admin queries and CSV exports run against a Postgres read replica, keeping reporting load off the transactional path (important while quizzes are live).

Item analysis deserves the precomputation: point-biserial correlation per question requires joining every answer with every attempt score. Done at close time for a 500-student quiz it is milliseconds; done live per dashboard load it would be the slowest query in the system.

Section 14

Security

  • Authentication. Short-lived JWT access tokens (15 min) + rotating refresh tokens in httpOnly cookies. First login with an admin-issued credential forces a password reset. Argon2id for hashing.
  • Authorization. Central can(actor, action, resource) policy invoked by every handler and by the realtime gateway at channel subscribe. Ownership checks (quiz owner, attempt owner, assignment) live in the policy, not scattered in handlers.
  • Answer-key isolation. The correct field is stripped by the serializer for any student-facing response; results endpoints expose it only after the quiz closes and the teacher releases results. This is enforced at the serialization layer so no new endpoint can leak it by accident.
  • Anti-abuse during attempts. Rate limits on answer saves (per attempt), single active session per student (starting an attempt from a second device invalidates the first socket and is logged as an event the teacher can see), and 404-not-403 responses for unassigned resources.
  • Guardrail trust model (v2). Violation reports originate in the client and are therefore spoofable in both directions - they are stored as evidence and surfaced to the teacher, never used as a sole automatic trigger beyond the configured ladder. Enforcement actions (kick, auto-submit) are server-side status flips that hold regardless of what the client does or fails to report.
  • Kick integrity (v2). The kick endpoint requires quiz ownership or admin role, a non-empty reason, and is rate-limited; the terminated attempt, its answers, and the reason are immutable after the fact, so a kick can be reviewed but never silently rewritten.
  • Auditability. Every mutation by admin or teacher lands in audit_log with actor, action, resource, and diff. Attempt events give the same trail for students.
  • Transport & storage. TLS everywhere, encrypted at rest, pre-signed URLs for uploads scoped to content type and size, PII limited to name + email.

Section 15

Scaling & tech stack

The load shape

The defining pattern is the synchronized start: a scheduled quiz means hundreds of students hitting "start" in the same minute, then a steady drizzle of autosaves, then a submit spike at the deadline. Design responses:

  • Attempt start is one small transaction plus a cached question snapshot (the snapshot JSON is cached in Redis at go-live, so 1,000 starts do not re-serialize the quiz 1,000 times).
  • Autosaves are single-row upserts on a primary key - they scale linearly and never contend with each other.
  • Grading and rollups are queue jobs, so the close-time spike is absorbed by the queue rather than the request path.
  • The API tier and realtime gateway are stateless (state lives in Postgres/Redis) and scale horizontally behind the load balancer; Redis pub/sub bridges gateway nodes so a teacher and their students need not share a node.

Capacity sanity check

2,000 concurrent students × 1 autosave/10 s ≈ 200 writes/s, plus ~250 coalesced events/s of fan-out. A single modest Postgres instance and one Redis node handle this with an order of magnitude of headroom; the horizontal story above exists for growth, not for day one.

Open questions for review

  • Should results release automatically at quiz close, or always require an explicit teacher action? (Current design: teacher toggle per quiz, default auto-release.)
  • Do we need per-question timing (each question individually timed) in v1, or is per-attempt duration enough?
  • Is email a hard requirement for account provisioning, or will the admin distribute printed credentials for younger students?
  • Should a kicked student's result be visible to them immediately with the kick reason, or withheld until the teacher releases results for the quiz?
